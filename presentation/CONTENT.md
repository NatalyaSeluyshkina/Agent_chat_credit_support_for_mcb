## Слайд 1 — Что и как мы оцениваем: категории и метрики

**Заголовок:** Как мы оцениваем агента

**Блок A. 8 категорий обращений (датасет 180 кейсов).** Каждая проверяет свой аспект:

| Категория | n | Что проверяет |
|---|---|---|
| `info` | 45 | Информационные ответы по продуктам/документам/процессам — работа RAG |
| `transactional` | 45 | Запросы про конкретного клиента — RAG + данные из БД |
| `escalation_sales` | 18 | Триггер эскалации «намерение оформить продукт» |
| `escalation_negative` | 18 | Триггер эскалации «негатив клиента» |
| `edge_no_data` | 18 | Корректный отказ по вопросам вне нормативки |
| `edge_conflict` | 9 | Коллизия общего и продуктового правила (приоритет — продуктовому) |
| `edge_manipulation` | 18 | Манипуляции, запрос чужих данных, prompt injection |
| `offtopic` | 9 | Вежливое закрытие нерелевантных тем |

**Блок B. 5 метрик качества:**

| Метрика | Что показывает |
|---|---|
| **category** | Точность классификатора (узел `classify`) против размеченной категории |
| **outcome** | Совпал ли тип результата: `info` / `calculation` / `escalation` / `rejection` / `clarification` |
| **escalation** | Для `escalation_*`: эскалировано ли с верным триггером |
| **citation** | Попал ли эталонный пункт нормативки в источники ответа. Только для citable-категорий (`info`, `transactional`, `edge_conflict`) |
| **judge** | LLM-судья (GigaChat Lite) оценивает соответствие `expected_behavior`: 0 / 0.5 / 1 |

**Сноска:** в поведенческих категориях (эскалации, отказы, offtopic) корректный ответ
не содержит цитаты нормативки — поэтому citation там не считается (помечено «—»).

## Слайд 2 — Структура RAG

**Заголовок:** RAG: поиск по нормативной базе

**Визуал:** схема `rag_structure.mmd` (большая, по центру).

**Текст-подписи рядом со схемой:**
- **Индексация (один раз):** 5 регламентов → section-aware чанкинг (сохраняем `doc_id`,
  `section_id`, тег `scope`) → эмбеддинги **GigaChat (1024-dim)** → векторный индекс
  **Chroma (HNSW, косинус)**.
- **Поиск (на каждом запросе):** запрос → эмбеддинг → `retrieve(query, k=8)` с опциональным
  `product_filter` → top-k чанков с тегами области действия.
- **Разрешение коллизий — на генерации:** при конфликте общего (`scope=general`) и
  продуктового (`scope=specific`) правила приоритет у продуктового; ретривер не
  переранжирует, выбор делает LLM по тегам.

## Слайд 3 — Метрики RAG (retrieval)

**Заголовок:** Качество поиска: recall@k

**Подзаголовок:** Метрика — `recall@k`: доля кейсов, где хотя бы один эталонный пункт
нормативки попал в top-k (эталон — ручная разметка `referenced_documents`).

**Главное число (крупно):**
> **Грунтинг по содержательным категориям: recall@5 = 78.8% · recall@8 = 86.9%**
> (в проде агент ретривит с k=8)

**Таблица (citable-категории — то, что реально мерит RAG):**

| Категория | recall@5 | recall@8 |
|---|---:|---:|
| info | 82.2% | 84.4% |
| transactional | 73.3% | 86.7% |
| edge_conflict | 88.9% | **100%** |
| **Итог (citable)** | **78.8%** | **86.9%** |

**Сноска:** поведенческие категории (`escalation_*`, `edge_no_data`, `edge_manipulation`,
`offtopic`) в итог не входят — там эталон указывает на регламент поведения агента, а не
на смысловой пункт; их обрабатывают узлы `classify`/`escalate`, а не RAG-поиск. Размытый
общий по всем 8 категориям (recall@8 = 64.4%) для оценки retrieval не используется.

## Слайд 4 — Тесты инструментов (tools) и графа

**Заголовок:** Детерминированные тесты: 4 слоя, 16 проверок — все зелёные ✅

**4 карточки (по слою):**

| Слой | Проверки | Что гарантирует |
|---|---|---|
| **1. Инструменты + БД** | auth · access control · запросы к SQLite · LangChain-обёртки · подбор продуктов | Идентификация по каналу; анонимный канал не получает данные; корректные профиль/кредиты/заявки; `get_eligible_products` не раскрывает скоринг; нет галлюцинаций на неизвестном клиенте |
| **2. Состояние + стык с RAG** | seed multi-turn · анонимный seed · редьюсер · RAG-стык | Инициализация состояния; `add_messages` копит историю диалога; формат совместим с RAG-хелперами (`build_query_from_history`, `detect_product`) |
| **3. Граф + безопасность** | маршрутизация · блок БД на анониме · multi-turn память · предохранитель | Верные ветки графа; checkpointer держит контекст диалога; запрос чужих данных/представителя → принудительная `suspicious`-эскалация, БД не тронута |
| **4. Эскалация** | пакет intent · пакет negative (multi-turn) · лимит сводки · лог графа | Сбор пакета оператору с историей; сводка ≤ 500 символов; эскалация проходит через граф и пишется в лог передачи |

**Внизу:** «Предохранители работают поверх LLM и не зависят от стохастики модели».

## Слайд 5 — Граф агента

**Заголовок:** Маршрутизация: граф LangGraph

**Визуал:** схема `agent_graph.mmd` (крупно, по центру).

**Текст-подписи:**
- `classify` — единая точка входа: категория + детекторы безопасности; дальше условные
  переходы.
- Ветки: эскалация/манипуляция → `escalate`; нужны данные клиента → `query_db`; нужна
  нормативка → `retrieve_rag`; не нужно ни то ни другое → сразу `generate_answer`.
- `query_db` может дополнительно уйти в `retrieve_rag` (транзакционный кейс = БД + RAG).
- `generate_answer` — ответ с грунтингом в источниках; `escalate` — пакет оператору.
